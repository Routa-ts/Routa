# v0 Spec: OpenAPI Scaffold

## Feature

Trail reads an OpenAPI `.yaml` or `.json` file and scaffolds Hono-backed route contracts, route-local Zod schemas, HTTP handler stubs, a manifest, and a baseline.

## Scope

In v0:

- OpenAPI input supports `.yaml` and `.json`.
- Generated schemas use Zod.
- Generated routes target the Hono adapter.
- `schemas.ts` is the recommended route-local schema file.
- Inline operation schemas generate route-local schemas by default.
- Explicit OpenAPI component `$ref` schemas keep shared identity.
- Matching inline schemas are not auto-shared without explicit component identity.
- Trail does not generate service files.

## Acceptance Cases

```yaml
case: v0_scaffold_yaml_basic_collection
intent: scaffold a collection route from OpenAPI YAML
input:
  files:
    openapi.yaml:
      paths:
        /users:
          get:
            operationId: listUsers
          post:
            operationId: createUser
action: trail scaffold openapi.yaml
expected:
  files:
    - routes/users/route.ts
    - routes/users/schemas.ts
    - .trail/manifest.json
    - .trail/openapi-baseline.json
    - .trail/routes.gen.ts
  behavior:
    - route file contains get and post contracts
    - route file imports route-local schemas
    - route handler contains an HTTP boundary stub or TODO
    - generated files include Trail comment header
    - manifest tracks route, schema, and route metadata files
    - baseline is created and not gitignored by Trail defaults
    - route metadata records resolved paths, methods, and middleware chains
must_not:
  - generate Fastify or Express code
  - generate non-Zod schemas
  - generate services/users.ts
  - require user to write OpenAPI twice
```

```yaml
case: v0_scaffold_json_basic_item
intent: scaffold an item route from OpenAPI JSON
input:
  files:
    openapi.json:
      paths:
        /users/{id}:
          get:
            operationId: getUserById
          patch:
            operationId: updateUser
action: trail scaffold openapi.json
expected:
  files:
    - routes/users/$id/route.ts
    - routes/users/$id/schemas.ts
  behavior:
    - path parameter id becomes route params schema
    - item route uses params plus method contracts
    - no service file is generated
must_not:
  - place item methods in routes/users/route.ts
  - generate or overwrite application business logic files
```

```yaml
case: v0_scaffold_rejects_unsupported_input_extension
intent: only yaml and json OpenAPI inputs are accepted
input:
  files:
    openapi.txt: "not supported"
action: trail scaffold openapi.txt
expected:
  behavior:
    - command fails before writing files
    - error identifies supported extensions
failure:
  mode: no files created
```

## Generated Route Shape

Generated route files should be readable and small.

Example shape:

```ts
// Generated by Trail.
// Safe to edit, but regeneration may update this file after preview.
// Source: openapi.yaml

import { defineRoute, createRoute } from "trail";
import { ListUsersQuery, UserListResponse } from "./schemas";
export default defineRoute({
	get: createRoute({
		input: {
			query: ListUsersQuery,
		},
		responses: {
			success: {
				status: 200,
				schema: UserListResponse,
			},
		},
		run: async ({ input, ctx }) => {
			// TODO: call application-owned business logic.
			return { type: "success", data: [] };
		},
	}),
});
```

## Out of Scope

- SDK generation.
- Docs UI generation.
- Runtime adapter choice.
- Auth provider generation.
