# Trail v0 Requirements

## Purpose

v0 exists to validate Trail as a real framework idea, not to deliver the full v1 vision.

The proof is:

```txt
openapi.yaml/json
-> route-local Zod schemas
-> Hono-backed route contracts
-> route handlers
-> developer-owned business logic
-> generated OpenAPI check
-> safe diff-based regeneration
```

If this loop is not clear, safe, and useful, Trail is not ready to grow into the broader v1 plan.

## Product Thesis

Trail v0 is for new TypeScript REST APIs.

The core value is not replacing existing frameworks. The core value is making a new API faster to start and easier to keep aligned:

- Schemas are the source of truth.
- OpenAPI can scaffold source.
- Source can generate OpenAPI.
- The HTTP boundary is explicit and typed.
- Business logic stays application-owned.

## v0 Scope

- Hono runtime adapter only.
- Zod schemas only.
- OpenAPI input files:
  - `.yaml`
  - `.json`
- OpenAPI-to-source scaffold.
- Source-to-OpenAPI generation.
- OpenAPI check against generated source.
- Project check command that runs Trail graph validation and TypeScript typechecking without emitting JavaScript.
- Build command that reuses Trail graph validation before compiling TypeScript to JavaScript.
- Layered internal testing strategy that validates types, graph checks, generation, CLI behavior, OpenAPI drift, runtime requests, diagnostics, regeneration safety, and guardrails.
- Safe regeneration through diff, preview, and confirmation.
- Route-local schema convention.
- Directory-style generated routes by default.
- Route parser designed so flat route support can be added without changing the route graph model.
- Minimal typed middleware surface only where needed to prove route-specific handler `ctx` inference.
- Global, group, resource segment, route-file, and per-method middleware inheritance.
- Generated `.trail/routes.gen.ts` route metadata for route tree, resolved middleware chains, and route-specific ctx types.

## Out of Scope for v0

- Fastify, Express, or other runtime adapters.
- Non-Zod schema adapters.
- Generated flat route output.
- Full mixed flat/directory routing support.
- Full auth provider system.
- Full security integration system.
- Rate-limit integration.
- Broad operations features.
- Full docs UI.
- Plugin system.
- SDK generation.
- Custom serializer registry.

## Default Project Layout

Recommended generated layout:

```txt
routes/
  middleware.ts
  users/
    middleware.ts
    route.ts
    schemas.ts
    /$id/
      route.ts
      schemas.ts

.trail/
  openapi-baseline.json
  manifest.json
  routes.gen.ts
```

v0 generated output uses directory-style routes only.

Future flat route support should resolve to the same route graph:

```txt
routes/users.ts        -> /users
routes/users.$id.ts    -> /users/:id
```

Mixed directory and flat routes should be allowed in the same project once flat support exists, but conflicting route definitions must fail build/check.

## Ownership Model

Trail-managed by default:

- `routes/**/route.ts`
- `routes/**/schemas.ts` when generated by Trail
- `.trail/openapi-baseline.json`
- `.trail/manifest.json`
- `.trail/routes.gen.ts`

User-owned by default:

- services, use cases, modules, policies, models, database code, and domain logic
- Custom schema files not generated or marked by Trail

Trail must not generate, overwrite, or assume ownership of application business logic files.

## Schemas

`schemas.ts` is the recommended route-local schema file name.

This is a convention, not lock-in.

Rules:

- Route contracts only care about imported schemas.
- Users may rename, colocate, centralize, or split schema files.
- Trail scaffold uses `schemas.ts` because it keeps the contract source close to the route.
- Trail regeneration only treats a schema file as managed if it was generated by Trail and tracked in the manifest, or if the user explicitly accepts that file in a preview.
- Inline operation schemas generate route-local schema declarations by default.
- OpenAPI component `$ref` schemas keep shared identity and should be imported or referenced as shared schemas.
- Schemas without explicit OpenAPI component identity must not be automatically shared across routes.
- Generated schema names should use contract-purpose names, such as `GetCurrentUserResponse`, `CreateUserBody`, or `CreateUserResponse`.

## Route Contracts and Middleware

Route contracts should infer handler types from declared schemas, responses, and middleware.

```txt
schemas -> handler input type
responses -> allowed return variants
middleware flow -> handler ctx type
```

Rules:

- Users should not need to manually annotate handler input.
- Handler `input` is inferred from route schemas, including params, query, headers, cookies, and body.
- Handler return values are constrained to declared response variants.
- Response variant `data` is typed from the declared response schema.
- Middleware is a typed pipeline, not only runtime functions.
- Middleware declares `requires`, `provides`, `rejects`, and the HTTP `input` it reads.
- Middleware-provided ctx belongs to the resolved route, not to one global application ctx.
- Trail should avoid a giant `AppCtx` that gives every route fields provided only by protected or admin middleware.
- Route-file middleware on `defineRoute` applies to every method in that route file.
- Per-method middleware on `createRoute` applies only to that HTTP method and runs after route-file middleware.

Middleware conventions:

```txt
routes/
  middleware.ts
  (private)/
    middleware.ts
    users/
      middleware.ts
      route.ts
  (admin)/
    middleware.ts
    reports/
      middleware.ts
      route.ts
```

Groups such as `(private)` and `(admin)` collect routes without adding URL segments.

Resolved order for `routes/(admin)/reports/route.ts`:

```txt
routes/middleware.ts
-> routes/(admin)/middleware.ts
-> routes/(admin)/reports/middleware.ts
-> routes/(admin)/reports/route.ts middleware
-> createRoute(...) middleware
-> handler
```

Middleware inheritance is resolved before handler ctx inference.

Trail stores resolved route metadata in:

```txt
.trail/routes.gen.ts
```

The generated route metadata should include route file, resolved path, URL-less groups, resource segments, methods, resolved middleware chain, and route-specific ctx types.

## Business Logic Boundary

Trail does not generate services.

Rules:

- The route handler lives in the route file.
- The handler maps validated HTTP-shaped input and typed context into application code.
- The developer decides whether application code uses services, modules, use cases, repositories, hexagonal architecture, feature folders, or any other structure.
- Generated route handlers may include TODO placeholders, but they should not create service files.
- Trail must not force a service architecture.

## Generated File Markers

Generated files should include a short comment header.

Example:

```ts
// Generated by Trail.
// Safe to edit, but regeneration may update this file after preview.
// Source: openapi.yaml
```

The comment is for humans. Trail must not rely only on comments to decide ownership.

## Manifest

Trail tracks generated files in:

```txt
.trail/manifest.json
```

The manifest should be committed to git.

It should track at least:

- Trail manifest version.
- OpenAPI baseline path.
- Generated file paths.
- Source OpenAPI file.
- Generated file kind.
- Related operation ids or schema names.
- Generated route metadata path.

Example:

```json
{
	"version": 1,
	"openapi": {
		"baseline": ".trail/openapi-baseline.json"
	},
	"generated": [
		{
			"path": "routes/users/route.ts",
			"source": "openapi.yaml",
			"operationIds": ["listUsers", "createUser"],
			"kind": "route"
		},
		{
			"path": "routes/users/schemas.ts",
			"source": "openapi.yaml",
			"schemas": ["User", "CreateUserInput"],
			"kind": "schemas"
		},
		{
			"path": ".trail/routes.gen.ts",
			"kind": "route-metadata"
		}
	]
}
```

## OpenAPI Baseline

Default baseline path:

```txt
.trail/openapi-baseline.json
```

The baseline should be committed to git.

Trail should make sure its own generated `.gitignore` defaults do not ignore this file.

The baseline supports:

- contract drift checks
- breaking-change checks
- regeneration diffs
- current state vs forward state comparison

## Regeneration

Regeneration must be stateful and diff-based, not blind overwrite.

Flow:

```txt
current baseline
+ forward OpenAPI input
-> diff
-> preview affected files
-> warn on overwrite/conflict
-> user confirms or aborts
-> apply accepted changes
-> update manifest
-> optionally update baseline
```

Rules:

- Preview changes before writing.
- Stop on conflicts by default.
- Warn when an existing route file will be changed.
- Warn when a generated file has manual edits.
- Preserve user-owned application files.
- Do not silently delete files.
- Require explicit confirmation before overwriting generated files with conflicts.

## v0 Commands

Required command concepts:

```txt
trail scaffold openapi.yaml
trail scaffold openapi.json
trail check
trail build
trail openapi check
trail openapi breaking
trail openapi breaking --update-baseline
```

Exact command names may change during implementation, but v0 must support these workflows.

Trail validation should use compiler-like graph infrastructure, not generic linting.

Reusable pipeline:

```txt
inputs
-> parse
-> route graph
-> schema graph
-> middleware graph
-> name registry
-> diagnostics
-> command output
```

`trail check` validates the Trail contract graph, then runs TypeScript typechecking without emitting JavaScript.

Trail graph validation should run before TypeScript so Trail can report framework-aware diagnostics instead of only generic TypeScript errors.

Trail graph validation should catch at least:

- duplicate schema names in the same generated or user-authored schema module
- invalid generated TypeScript identifiers
- missing or duplicate `operationId` values where applicable
- route-local schema import and name conflicts
- two route files resolving to the same path and method
- invalid path params
- middleware requiring ctx fields not provided by earlier middleware
- missing request or response schemas where required by Trail contracts
- `$ref` values that point to missing OpenAPI components
- unsupported OpenAPI constructs in v0
- manual edits or manifest drift in generated files where detectable
- stale or inconsistent `.trail/routes.gen.ts` route metadata where detectable

`trail check` should also catch TypeScript type errors through `tsc --noEmit` or an equivalent typecheck.

`trail build` validates the same Trail contract graph, then compiles TypeScript to JavaScript.

`trail build` should reuse Trail graph validation, but should not need to run a separate `tsc --noEmit` pass before TypeScript emit.

If Trail graph validation fails, `trail build` must fail before emitting build output.

`trail openapi check` remains the focused OpenAPI drift and contract-quality command. `trail check` may call or include its relevant checks, but the top-level command is the default project validation command.

## Validation Requirements

OpenAPI input checks should warn or fail, depending on mode, for:

- unsupported file type
- invalid OpenAPI document
- missing or duplicate `operationId`
- missing request or response schemas
- missing path parameter schemas
- unsupported media types for v0
- names that cannot become stable TypeScript identifiers
- ambiguous same-status responses
- security metadata that cannot map to Trail placeholders

## Success Criteria

v0 is successful if a proof of concept can:

1. Read an OpenAPI `.yaml` or `.json` file.
2. Generate Hono-backed Trail route files.
3. Generate route-local Zod schemas by convention.
4. Generate route handlers that leave business logic to the developer.
5. Generate and commit `.trail/manifest.json`.
6. Generate and commit `.trail/openapi-baseline.json`.
7. Regenerate from a changed OpenAPI input through a previewed diff.
8. Preserve application business logic files during regeneration.
9. Generate OpenAPI from source and check it against expected output.
10. Run `trail check` without emitting JavaScript.
11. Run `trail build` so it emits JavaScript only after Trail graph validation passes.
12. Infer route handler input, return variants, and ctx from schemas, responses, and resolved middleware.
13. Generate `.trail/routes.gen.ts` with route tree and route-specific middleware ctx metadata.
14. Prove the v0 loop with the layered internal testing matrix in `testing_strategy.md`.

## Green Light

Trail v0 gets a green light only if it proves:

```txt
OpenAPI .yaml/json
-> route-local Zod schemas
-> Hono route contracts
-> route handlers
-> developer-owned business logic
-> committed baseline and manifest
-> safe regeneration
```

Without safe regeneration, OpenAPI-to-source is not trustworthy enough to be Trail's core.
