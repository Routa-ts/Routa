# Routa Governance

Routa uses lightweight maintainer-led governance while the project is young.

## Roles

### Users

Users install Routa, open issues, ask questions, and report bugs. Users do not need to contribute code to participate.

### Contributors

Contributors improve Routa through issues, pull requests, docs, examples, tests, design feedback, or community support.

### Triagers

Triagers help label issues, request reproductions, confirm bugs, close duplicates, and route discussions. Triagers do not merge code unless they are also maintainers.

### Maintainers

Maintainers review and merge PRs, manage releases, resolve issue priority, and make final decisions for Routa's public API and direction.

### Owner

The owner is responsible for repository access, npm publishing access, security escalation, and final conflict resolution.

## Decision Process

- Small fixes can be proposed directly in pull requests.
- Public API changes, large features, and behavior changes should start as an issue or discussion.
- Maintainers make final calls after considering technical fit, maintenance cost, compatibility, and v0 scope.
- Decisions can be revisited when new evidence appears, but merged behavior should not be churned casually.

## Becoming a Maintainer

Maintainers may invite contributors who repeatedly show good judgment, useful reviews, reliable follow-through, and care for the project's long-term health.

## Releases

Routa uses Changesets and fixed versions across the public packages. See [Release](./docs/release.md).

## Conduct

All participation is covered by the [Code of Conduct](./CODE_OF_CONDUCT.md).
